SupplyTrack: A Sustainable Procurement and Supply Chain Management System
Database Systems Course Project Report

Group Members: Praneeth Rangarajan, Rithwak Somepalli

────────────────────────────────────────────────────────────────────────────────

1. APPLICATION OVERVIEW AND MOTIVATION

SupplyTrack is a web-based procurement platform that allows buyers to browse
products from global suppliers, compare shipping routes optimized by cost, speed,
and carbon footprint, and place orders directly through the interface. Suppliers
receive a parallel dashboard to manage their product catalog, inventory levels, and
incoming orders. The application is built to demonstrate how responsible logistics
decisions can be supported through careful database design and analytical computation.

We chose this application because supply chain management is inherently relational:
it involves many-to-many supplier-product relationships, nested order structures, and
complex multi-table queries, making it an ideal vehicle for applying database design
principles. Most comparable systems simply present buyers with one shipping option.
SupplyTrack distinguishes itself by giving buyers measurable trade-off data — cost,
delivery time, and carbon emissions — across multiple route alternatives. This
sustainability angle is timely, commercially relevant, and technically interesting
in ways that a simpler project (such as a blog or student portal) would not be.


2. KEY COMPONENTS

SupplyTrack consists of four major components. The Product Catalog surfaces all
in-stock products with their associated suppliers and pricing, drawn from a
three-table join across product, supplier_product, and supplier. The Order System
handles the full procurement lifecycle: stock validation, order creation, inventory
depletion, and order history retrieval for both buyers and sellers. The Transport
Optimization Engine (the advanced function) generates and ranks multi-leg shipping
routes between the supplier's city and the buyer's destination. Finally, an
Interactive Route Map renders each route as a polyline on an embedded Leaflet.js
map, allowing buyers to visually compare routes before confirming their order.


3. DATABASE DESIGN

3.1 E-R Model

The core entities are: PRODUCT (goods sold on the platform), SUPPLIER (companies
that carry those goods), USER (buyers), TRANSPORT_METHOD (shipping modes such as
cargo flight, freight truck, sea freight, or electric van), CARBON_GOAL (an
organizational sustainability record), ORDER (a buyer's procurement transaction),
and ORDER_ITEM (a line item within an order).

The central relationships are: a SUPPLIER offers many PRODUCTs and a PRODUCT is
offered by many SUPPLIERs, resolved through the SUPPLIER_PRODUCT junction entity
which carries the attributes price, priority, and stock_qty. A USER places many
ORDERs; each ORDER belongs to one USER. An ORDER uses one TRANSPORT_METHOD. An
ORDER contains one or more ORDER_ITEMs, each of which references a valid
(supplier_id, product_id) pair from SUPPLIER_PRODUCT.

3.2 Relational Model and Tables

  PRODUCT(product_id PK, name, description, category, weight_kg)
  SUPPLIER(supplier_id PK, name, contact_email, country, city)
  USERS(user_id PK, name, email UNIQUE)
  TRANSPORT_METHOD(transport_id PK, mode, avg_speed_kmh, co2_per_km_kg)
  CARBON_GOAL(goal_id PK, period, target_co2, actual_co2)
  SUPPLIER_PRODUCT(supplier_id PK/FK, product_id PK/FK, price, priority, stock_qty)
  ORDERS(order_id PK, user_id FK, transport_id FK, order_date, total_price,
         status, ship_addr)
  ORDER_ITEM(order_id PK/FK, item_id PK, product_id, supplier_id, quantity,
             unit_price, item_co2_kg, FK(supplier_id, product_id) → SUPPLIER_PRODUCT)

All primary keys are UUID values generated by PostgreSQL's uuid_generate_v4()
function, chosen to avoid predictable key enumeration and to support future
distributed deployment.


4. NORMALIZATION, FUNCTIONAL DEPENDENCIES, AND CONSTRAINTS

Every table satisfies both Third Normal Form (3NF) and Boyce-Codd Normal Form
(BCNF). In PRODUCT, SUPPLIER, USERS, TRANSPORT_METHOD, and CARBON_GOAL, a single-
attribute primary key directly determines all other attributes, with no partial or
transitive dependencies. SUPPLIER_PRODUCT has a composite primary key
(supplier_id, product_id); price, priority, and stock_qty depend on the full
composite key, not on either attribute alone, satisfying BCNF. In ORDERS, order_id
determines all other columns. In ORDER_ITEM, the composite key (order_id, item_id)
determines all remaining attributes.

A notable design decision was placing supplier_id and product_id directly in
ORDER_ITEM, which enables enforcement of a composite foreign key
FK(supplier_id, product_id) → SUPPLIER_PRODUCT at the database level. This
guarantees that no order item can reference a supplier-product pairing that does
not exist — an integrity guarantee that would be unreliable if enforced only in
application code.

Additional constraints include: NOT NULL on all required fields; a UNIQUE constraint
on users.email; ON DELETE CASCADE on ORDER_ITEM → ORDERS and on
SUPPLIER_PRODUCT → SUPPLIER/PRODUCT; ON DELETE RESTRICT on ORDER_ITEM →
SUPPLIER_PRODUCT (protecting historical order integrity); ON DELETE SET NULL on
ORDERS → USERS and ORDERS → TRANSPORT_METHOD; and DEFAULT values for order_date
(now()), status ('pending'), and quantity (1). Performance indexes are created on
orders.user_id, orders.transport_id, order_item.order_id, and
supplier_product.product_id to accelerate the most common join patterns.


5. BASIC FUNCTIONS AND IMPLEMENTATION

The basic functions cover standard CRUD operations across the platform. The catalog
endpoint (GET /api/storefront/catalog) performs a multi-table join across product,
supplier_product, and supplier, filtering out any product with zero total stock.
User lookup (GET /api/storefront/users) retrieves registered buyers. Order placement
(POST /api/storefront/order) executes a three-step write: insert into orders, insert
into order_item, and decrement stock_qty in supplier_product. Buyer order history
(GET /api/buyer/orders/{user_id}) fetches orders with nested order item and product
data in a single batched query. Seller inventory management endpoints allow suppliers
to add products (POST /api/seller/product), link products to their catalog with
pricing (POST /api/seller/inventory), and zero out stock to delist items
(DELETE /api/seller/inventory/{supplier_id}/{product_id}). Sellers can also view
all active orders for their products (GET /api/seller/orders/{supplier_id}) and
dismiss fulfilled orders (POST /api/seller/orders/{order_id}/dismiss).


6. ADVANCED FUNCTION: TRANSPORT OPTIMIZATION ENGINE

The Transport Optimization Engine (app/services/transport_engine.py) dynamically
generates, evaluates, and ranks multi-leg shipping routes between a supplier's city
and the buyer's destination, based on the buyer's stated preference.

The engine maintains a coordinate database of cities across Asia, Europe, the
Americas, Africa, and the Middle East, along with a set of transit hub corridors
(e.g., Asia-to-Americas routes passing through Shanghai or Tokyo, then Los Angeles).
Given an origin and destination, it generates up to eight candidate waypoint
sequences, including the direct route and several one- and two-hub alternatives
chosen from corridors relevant to the geographic regions involved.

For each candidate route and transport method, the engine computes individual legs
using the Haversine formula to determine great-circle distance, then calculates
estimated travel time (distance divided by average speed, rounded up to whole days
with a handling buffer), shipping cost (a base fee plus a per-km rate scaled by
cargo weight with mode-specific caps, e.g., bicycles are limited to 500 km and
trucks cannot cross oceans), and carbon emissions (distance × CO2 factor × cargo
weight). Legs that are infeasible for a given transport mode invalidate the entire
route for that mode. The engine additionally generates mixed-transport routes for
multi-leg options, assigning air freight to long intercontinental legs and ground
transport to shorter inland segments.

All valid routes are then ranked using a normalized composite score. Four preference
modes are supported: "fastest" (70% weight on time), "cheapest" (70% weight on
cost), "greenest" (70% weight on CO2), and "balanced" (equal thirds). Each metric
is normalized against the maximum observed value across all candidates to ensure
fair comparison. Routes are sorted in ascending score order, and the top six are
returned. The highest scorer is tagged "Recommended"; individual best performers
receive "Fastest," "Eco-Friendly," or "Cheapest" tags; the rest are "Standard."

The results are displayed on a Leaflet.js map where each route is drawn as a
polyline connecting its waypoint cities, with markers at each stop. Route summary
cards show mode, distance, estimated days, cost, and CO2. Selecting a card
highlights the corresponding route on the map.


7. LANGUAGES, PLATFORM, AND FRONTEND-BACKEND INTERACTION

The backend is written in Python using FastAPI, chosen for its request validation
via Pydantic models, automatic API documentation, and strong async performance.
The database is PostgreSQL hosted on Supabase, which provides a managed cloud
instance and a Python client library for structured querying without raw SQL strings.
The frontend is vanilla HTML, CSS, and JavaScript — kept lightweight intentionally —
with Leaflet.js for the interactive map. The application launches via a start.bat
script that installs dependencies and starts a uvicorn server that serves both the
API and all static files from the same port.

The frontend communicates with the backend entirely through REST API calls using
the browser's native fetch() API. When a buyer clicks "View Routes," the frontend
calls GET /api/storefront/routes/{product_id}/{supplier_id} with the buyer's
preference and destination city as query parameters. The response JSON contains
geometric leg data (coordinates, distances) and summary statistics, which the
frontend uses to draw polylines on the map and populate the route cards. On order
confirmation, the frontend posts the selected transport ID and pre-computed CO2
and cost values to POST /api/storefront/order.


8. LESSONS LEARNED AND FUTURE EXTENSIONS

This project reinforced that early schema decisions have lasting consequences.
The most significant was the choice to embed supplier_id and product_id directly
in ORDER_ITEM to enable a composite foreign key constraint — a pragmatic tradeoff
that prioritized database-level integrity over strict adherence to the original E-R
model. We also learned the value of keeping computation separate from persistence:
routes are generated on-demand and never stored, avoiding cache invalidation
complexity and keeping the schema lean.

The hardest problem was resolving circular referential constraints introduced by an
early "involves" relationship table that linked orders, suppliers, and products.
The solution was to remove that table and fold the identifiers directly into
ORDER_ITEM, simplifying both the schema and the query paths.

In the future, SupplyTrack could be extended with JWT-based user authentication, a
route_leg table to persist multi-leg journeys for historical analytics, real
emission factor APIs replacing the simplified linear model, demand forecasting for
sellers, automated carbon goal alerts when actual CO2 approaches organizational
targets, and cloud deployment for public accessibility and horizontal scaling.
